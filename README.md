# **“灵犀”项目全栈开发纪要**

## **1\. 项目架构与技术选型**

* **系统架构图**：  
  * **感知层**：MAX30102 光电容积脉搏波（PPG）传感器，基于 I2C 协议实时采集高频红外（IR）与红光原始环境及血流反射数据。  
  * **主控层**：STM32F103C6T6 微控制器。作为核心调度枢纽，负责外设驱动、软 I2C 通信、时钟树配置、系统级硬件抽象，以及运行自研的抗噪滤波、自适应阈值解算与波形重构算法。  
  * **交互层**：0.96 英寸 OLED 显示屏（I2C 接口）与 PC13 板载 LED。OLED 负责多态文本状态机反馈及实时 PPG 波形扫描渲染；PC13 LED 作为“心跳包”与脉搏捕获的物理级微秒响应指示。  
* **核心技术栈及选型理由**：  
  * **主控 MCU (STM32F103C6T6)**：选用该芯片主要出于成本考量及基础引脚兼容性。其物理限制（仅 32KB Flash 与 10KB RAM）成为架构设计的核心瓶颈，倒逼项目摒弃了原厂高开销的数学计算库，转向内存空间复杂度为 $O(1)$ 的轻量级自研算法。  
  * **开发与构建链**：VS Code \+ EIDE 扩展 \+ ARM Compiler 5.06 (Keil MDK)。舍弃重量级 IDE，实现项目文件的结构化解耦与快速增量构建，便于精准排查编译链依赖问题。  
  * **驱动框架**：STM32 标准外设库（SPL）。寄存器级封装成熟，便于对 GPIO 进行高频翻转控制以及总线时序的微调。

## **2\. 核心算法与演进**

* **核心算法**：基于 PPG 信号的极简动态自适应峰值检测、防抖拦截与实时扫描式波形渲染算法。  
* **算法迭代优化过程**：  
  * **V1 官方重负载模型**：引入 Maxim 官方 maxim\_heart\_rate\_and\_oxygen\_saturation 库。需维护两个长度 500 的 32 位整型数组，RAM 开销超 4KB，直接导致系统 HardFault。  
  * **V2 静态差分检测**：彻底移除官方库。采用定值阈值与上一帧绝对比对（ir\_val \< last\_ir）。抗干扰极差，环境光及物理按压导致状态机高频震荡。  
  * **V3 迟滞比较防抖**：建立不对称双阈值（入场 60000 且持续 5 帧，出场 40000 且持续 10 帧），彻底解决临界信号的逻辑抖动。  
  * **V4 动态自适应与死区拦截（核心计算态）**：每 10 次轮询刷新 ir\_max 与 ir\_min 生成自适应基线，消除按压力度造成的直流偏置；引入 300ms 生理响应死区（Refractory Period）屏蔽锯齿波重影误判；统计窗口由 1 秒扩展至 3 秒进行数据平滑。  
  * **V5 动态归一化渲染（视觉呈现态）**：使用 (ir\_val \- ir\_min) / (ir\_max \- ir\_min \+ 1\) \* 35 映射物理坐标。废弃全屏清理指令，改用线段插值（OLED\_DrawLine）配合前置黑场扫描清屏，实现监护仪级丝滑波形。

## **3\. 问题与破局路径库**

* **问题 1：供电回路误区与参考资料谬误导致物理层瘫痪**  
  * **触发场景**：初期硬件接线阶段，设备无法正常上电与通讯。  
  * **尝试与失败**：依赖 AI（Copilot 与 Gemini）的常规通用建议进行排查，与实际非标准开发板布线错位；参考网络开源接线图，发现参考图本身存在引脚定义错误。  
  * **最终解决方案**：摒弃二手资料，直接通过物理短接引脚测试连通性，理清真实电路走向，最终实现供电回路物理级通畅。  
* **问题 2：引导模式配置错误导致固件烧录拒绝**  
  * **触发场景**：向单片机下发固件时，终端抛出 DEV\_TARGET\_RESET\_ERR 目标复位失败错误。  
  * **尝试与失败**：盲目测试 Reset 按钮，将 BOOT0 跳线帽保持在电平 '1' 状态进行烧录及运行测试，未能切断系统引导装载程序。  
  * **最终解决方案**：理清 STM32 启动逻辑，调换跳线帽将 BOOT0 引脚电平拉低至 '0'，使复位后程序指针重新指向主闪存（Main Flash）运行模式，解除死锁。  
* **问题 3：IDE 默认编译配置缺失导致无输出产物**  
  * **触发场景**：编译过程未报错，但工程目录下未生成可供烧录的二进制固件。  
  * **尝试与失败**：多次重复编译代码，未能改变输出结果。  
  * **最终解决方案**：进入编译工具链深层设置，手动勾选并配置生成 .bin 或 .hex 二进制文件的编译选项。  
* **问题 4：头文件路径与依赖树缺失导致编译中断**  
  * **触发场景**：代码引入算法逻辑后，编译器报 Error: \#5: cannot open source input file "algorithm.h"。  
  * **尝试与失败**：工程结构中未独立抽离该头文件，保留 \#include "algorithm.h" 语句无法通过依赖图谱校验。  
  * **最终解决方案**：删除 main.c 顶部的头文件引用，直接在代码顶部使用 extern 关键字显式声明核心算法的函数签名，交由链接器自动寻址。  
* **问题 5：项目文件混淆引发编译器“伪更新”**  
  * **触发场景**：重写主循环尝试点亮 PC13，编译烧录提示成功，但物理单片机仍执行旧版本代码逻辑。  
  * **尝试与失败**：反复修改当前 IDE 视窗内的 main.c，因本地同级目录残留 main(1).c 等副本，编译器优先级判定错乱，持续覆写历史文件。  
  * **最终解决方案**：物理销毁文件系统中的全部冗余 .c 副本，利用 Clean 指令擦除中间件，从最基础的“点灯”逻辑开始，强制插入 RCC\_APB2PeriphClockCmd 时钟使能语句验证固件更新有效性。  
* **问题 6：硬件中断阻塞引发主程序假死（死循环）**  
  * **触发场景**：OLED 初始化并显示文本后，PC13 停滞，系统不对传感器输入作出响应。  
  * **尝试与失败**：因无虚拟串口模块，无法通过打印日志定位死锁。  
  * **最终解决方案**：定位并彻底删除 while(MAX30102\_INT \== 1); 中断死锁阻塞逻辑。将被动的硬件中断等待降级为软件主动轮询读取机制，并在循环尾部补偿延时确保 I2C 总线时序完整。  
* **问题 7：内存资源耗尽触发系统 HardFault 异常**  
  * **触发场景**：消除阻塞并恢复官方心率算法调用后，PC13 蓝灯熄灭，系统进入不可逆硬件失效状态。  
  * **尝试与失败**：修改启动汇编文件 .s，尝试将 Stack\_Size 从 0x400 依次拉升至 0x800、0x1000；缩编数据缓冲数组容量至 200 并调整为静态全局分布。均因 C6T6 芯片仅有 10KB RAM 的极限物理天花板宣告失败。  
  * **最终解决方案**：彻底废除官方重负载算法调用。将堆栈降回安全值 0x400，自行重写完全无数组开销、纯依赖静态标量存储当前极值的轻量级检测逻辑。  
* **问题 8：临界边缘信号引发系统状态高频震荡**  
  * **触发场景**：手指初贴合传感器时，OLED 屏幕在“计算”与“搜寻”状态间高频横跳。  
  * **尝试与失败**：采用单一绝对阈值 \> 50000 判断，环境漫反射及微小物理形变导致信号频繁击穿该单向屏障。  
  * **最终解决方案**：引入迟滞比较器逻辑。建立高阈值线（60000，连续 5 帧确认入场）与低阈值线（40000，连续 10 帧确认出场），构建宽频抗扰容错带。  
* **问题 9：静态阈值阻断与时间窗口窄化导致心率失真**  
  * **触发场景**：系统稳态进入 Sensing，但长时间不输出数字，或输出极度失真的数值波动。  
  * **尝试与失败**：设定 \>80000 固定判定值（手指重压导致波形平坦化无法触发）；采用 1 秒观测窗口（单次硬件级抖动误判乘以 60 被放大为极其离谱的误差）。  
  * **最终解决方案**：部署动态跟踪基线，每 10 次轮询计算自适应阈值；引入 300ms 物理死区拦截高频噪点重影触发；特征统计窗口拉长至 3 秒，配合 45-180 的合理生理区间过滤器拦截超限乱码。  
* **问题 10：链接错误与空声明警告阻碍固件生成**  
  * **触发场景**：集成波形绘制代码时，触发 L6218E: Undefined symbol OLED\_ClearLine 致命链接错误及大量未引用警告。  
  * **尝试与失败**：尝试调用底层库中仅在 .h 声明但未在 .c 实现的局部清理 API。  
  * **最终解决方案**：剥除 OLED\_ClearLine 逻辑层。重构波形防抖逻辑以激活废弃的计时变量消解警告。替换绘图底层 API 为标准化兼容度极高的 OLED\_DrawPoint。  
* **问题 11：状态机逻辑孤岛导致渲染层挂起**  
  * **触发场景**：固件无报错烧录成功，但 OLED 永久定格在 "Standby" 等待页，拒绝切入波形渲染态。  
  * **尝试与失败**：主轮询中孤立执行 if(finger\_status \== 1\) 判断，但前置将该标志位置 1 的物理环境探测层被意外架空。  
  * **最终解决方案**：在逻辑树最外层追加 if(ir\_val \> 55000\) 环境鉴别器，在其内部作用域完成 finger\_status 的状态置位、原点重置及清屏调度，闭环逻辑链。  
* **问题 12：绘图散点离散与全屏清理引发总线通信卡顿**  
  * **触发场景**：绘制出初始波形，但曲线表现为断裂的散点；横轴扫描触底换行时，系统出现宏观掉帧卡滞。  
  * **尝试与失败**：基于单点坐标绘制；利用 OLED\_Clear() 响应越界回车，此指令在软 I2C 下引发的 1024 字节全量覆写导致总线严重阻塞。  
  * **最终解决方案**：采用 OLED\_DrawLine 连接历史与当前坐标进行插值补线；废除整屏格式化，采用“前置黑场扫描”方案，在推进坐标前预先写入 X+1 与 X+2 列的黑场像素（显存归零），彻底释放 I2C 通信带宽。
